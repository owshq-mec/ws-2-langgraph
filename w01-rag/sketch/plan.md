# DataOps Knowledge Hub — Project Plan

## Vision

Build an **Enterprise RAG system** that demonstrates how Retrieval-Augmented Generation works across multiple data stores in a real-world data engineering environment. This is not a toy example with a single PDF — it is a production-grade system that queries structured, semi-structured, and unstructured data through intelligent routing.

The system answers complex cross-domain questions by decomposing them into sub-queries, routing each to the appropriate data store, and returning validated structured output.

---

## Architecture: Ledger + Memory + Brain

The Knowledge Hub divides organizational knowledge into three cognitive domains:

| Domain | Role | Data Store | LlamaIndex Engine | Query Type |
|--------|------|-----------|-------------------|------------|
| **Ledger** | The factual truth — numbers, transactions, metrics | PostgreSQL | `NLSQLTableQueryEngine` | Text-to-SQL |
| **Memory** | Historical context — documents, logs, policies | Qdrant | `VectorStoreIndex` | Semantic Vector Search |
| **Brain** | Connections — relationships, lineage, dependencies | Neo4j | `PropertyGraphIndex` | Cypher Graph Traversal |

A `SubQuestionQueryEngine` (or `RouterQueryEngine`) sits on top, decomposing complex questions and routing sub-queries to the appropriate engine.

## Data Sources (Generated by Data Generator)

### PostgreSQL (Ledger)

```sql
-- customers
CREATE TABLE customers (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100),
    plan VARCHAR(20),        -- free, pro, enterprise
    company VARCHAR(100),
    created_at TIMESTAMP
);

-- orders
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    customer_id INTEGER REFERENCES customers(id),
    product_id INTEGER REFERENCES products(id),
    amount DECIMAL(10,2),
    status VARCHAR(20),      -- pending, completed, failed
    created_at TIMESTAMP
);

-- products
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100),
    category VARCHAR(50),
    price DECIMAL(10,2),
    sku VARCHAR(20)
);
```

## Success Criteria

The project is **done** when:

1. `docker compose up` brings all services healthy in under 60 seconds
2. Data Generator populates all 3 sources continuously
3. A single question like *"Which enterprise customers spent over R$50k, had pipeline incidents, and what downstream systems would be impacted?"* returns a valid Pydantic-validated JSON response that includes data from all 3 stores
4. The FastAPI `/docs` shows interactive Swagger documentation
5. Claude Code can query the system via MCP in production
6. The entire system is deployed and accessible via public URL

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Vector DB | Qdrant (not ChromaDB) | Production-grade, filtering, hybrid search |
| Graph DB | Neo4j (not Memgraph) | Native LlamaIndex integration (`Neo4jPropertyGraphStore`) |
| Object Storage | SeaweedFS (not MinIO) | Active project, S3-compatible, lightweight |
| Chunking | SemanticSplitter | Better retrieval quality than fixed-size |
| Router | SubQuestionQueryEngine | Handles multi-domain questions natively |
| API | FastAPI | Native Pydantic integration, async, auto-docs |
| Deploy | Railway | Docker Compose native, 1-click, visual logs |
